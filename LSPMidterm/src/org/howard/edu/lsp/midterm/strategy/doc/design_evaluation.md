The original implementation of the PriceCalculator class is poorly designed because it relies on a long if-else chain to handle different customer types. This violates the Open-Closed Principle, which says that a class should be open for extension but closed for modification. In the old version, if you wanted to add a new "Student Discount" or "Senior Citizen" rate, you would have to go inside the PriceCalculator and manually change the existing, tested code.

By refactoring this to the Strategy Pattern, we've separated the math for each discount into its own separate class. Now, PriceCalculator doesn't need to know how the discount is calculated; it just knows it has a "strategy" that can handle it. This makes the system much more flexible and easier to maintain because you can add as many new pricing types as you want without ever touching the core calculator logic again.
